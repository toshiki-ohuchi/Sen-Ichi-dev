# 7. 共通仕様

## 7.1 エラーハンドリング方針

### バックエンド（API）

| エラー種別 | 対応方針 |
|---|---|
| バリデーションエラー | `400` + エラーメッセージをレスポンスに含める |
| 認証エラー | `401` または `403` |
| リソース未存在 | `404` |
| 競合 | `409` + `{ error: 'conflict', message: '...' }` |
| サーバーエラー | Hono のデフォルトエラーハンドラ（`500`） |

### フロントエンド（Vue）

| エラー種別 | 表示方法 |
|---|---|
| API通信エラー | `alert()` でエラーメッセージを表示 |
| 楽観的ロック競合（409） | 「他のユーザーがこのレコードを更新しました。画面を更新してください。」を alert で表示 |
| ログイン失敗 | フォーム下部にエラーメッセージを表示 |
| ネットワークエラー | コンソールにエラー出力（UIへの通知は最低限） |

## 7.2 楽観的ロック仕様

複数ユーザーが同一レコードを同時に編集した場合の競合を検出・防止する仕組みです。

### 仕組み

```
ユーザーA                      ユーザーB                  DB
  │                              │                       │
  ├─ GET /records/1 ────────────────────────────────────▶│
  │◀─ { version: 3 } ──────────────────────────────────  │
  │                              │                       │
  │                    ├─ GET /records/1 ────────────────▶│
  │                    │◀─ { version: 3 } ──────────────  │
  │                              │                       │
  ├─ PUT /records/1 ────────────────────────────────────▶│
  │   { version: 3 }             │           version:3==3│
  │◀─ 200 OK (version=4) ───────────────────────────────  │
  │                              │                       │
  │                    ├─ PUT /records/1 ────────────────▶│
  │                    │   { version: 3 }    version:3≠4  │
  │                    │◀─ 409 Conflict ─────────────────  │
  │                              │                       │
```

### versionフィールド仕様

| 項目 | 仕様 |
|---|---|
| 初期値 | `1`（新規登録時） |
| 更新時 | `clientVersion + 1`（毎更新でインクリメント） |
| 競合検出条件 | リクエストの `version` ≠ DB上の `version` |
| 競合時のHTTPステータス | **409 Conflict** |
| 競合時のレスポンス | `{ "error": "conflict", "message": "他のユーザーがこのレコードを更新しました。画面を更新してください。" }` |
| クライアントの対応 | alertで通知し、ユーザーに画面更新を促す |

## 7.3 ページネーション仕様

### クエリパラメータ

| パラメータ | デフォルト | 説明 |
|---|---|---|
| `page` | `1` | ページ番号（1始まり） |
| `pageSize` | `20` | 1ページあたりの件数 |

### レスポンス形式

```json
{
  "data": [...],
  "total": 150,
  "page": 2,
  "pageSize": 20
}
```

### オフセット計算

```
offset = (page - 1) * pageSize
```

例：`page=2, pageSize=20` の場合、`offset=20`（21件目から取得）

## 7.4 ソート仕様

### ソートキー

| キー | 対応フィールド | デフォルト |
|---|---|---|
| `no` | `records.no` | ○（デフォルト） |
| `customerName` | `records.customer_name` | - |
| `revenue` | `records.revenue` | - |
| `itPartnerRatio` | `records.it_partner_ratio` | - |
| `sasRatio` | `records.sas_ratio` | - |
| `memberCount` | `records.member_count` | - |
| `avgUnitPrice` | `records.avg_unit_price` | - |
| `updatedAt` | `records.updated_at` | - |

### ソート方向

| 値 | 説明 |
|---|---|
| `asc` | 昇順（デフォルト） |
| `desc` | 降順 |

### フロントエンドのソート切替ロジック

```typescript
function setSort(key: string) {
  if (query.sortKey === key) {
    // 同じキーで再クリック → 方向を反転
    query.sortDir = query.sortDir === 'asc' ? 'desc' : 'asc'
  } else {
    // 別のキー → 昇順でリセット
    query.sortKey = key
    query.sortDir = 'asc'
  }
  query.page = 1  // 1ページ目に戻る
  fetchList()
}
```

## 7.5 フィルタ仕様

### 検索・フィルタの組み合わせ

複数条件はAND条件で結合されます。

```sql
WHERE
  (customer_name ILIKE '%keyword%' OR project_name ILIKE '%keyword%')
  AND assigned_dept = 'ITコンサルティング部'
  AND commercial_flow = '1次'
  AND contract_type = '請負'
  AND customer_grade = 'A'
```

### フィルタ選択肢の取得

`GET /api/records/filters` でDB上の実データをDISTINCTで取得。データ追加・更新に応じて動的に変化します。

対象フィールド：`assignedDept` / `commercialFlow` / `contractType` / `customerGrade`

### フィルタリセット

`resetFilters()` で全条件をクリアし、`page=1` に戻してデータを再取得します。

## 7.8 月次スケジュールの保存仕様

### データ構造

月次スケジュールは「活動種別 × 月」の組み合わせで1レコードとしてDB保存します。

```
records.id=1 のスケジュール例：
  { recordId:1, month:'3月', activityType:'顧客イベント', content:'FY26 1Q' }
  { recordId:1, month:'4月', activityType:'顧客イベント', content:'FY26 2Q' }
  { recordId:1, month:'3月', activityType:'SI計画（作業時間）', content:'芦田参画終了' }
  ...
```

### 更新方式（Upsert）

レコード更新時、スケジュールは全セル分を Upsert（ON CONFLICT DO UPDATE）で保存します。

```sql
INSERT INTO monthly_schedules (record_id, month, activity_type, content)
VALUES (?, ?, ?, ?)
ON CONFLICT (record_id, month, activity_type)
DO UPDATE SET content = EXCLUDED.content
```

### フロントエンドの内部状態管理

`TabSchedule.vue` では `scheduleMap`（`Record<activityType, Record<month, content>>`）という2次元Mapで管理し、フォームへの同期時に配列形式に変換します。

`isSyncing` フラグを使用して `watch` の無限ループを防止しています。
