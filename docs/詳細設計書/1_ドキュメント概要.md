# 1. ドキュメント概要

## 1.1 目的

本ドキュメントは、「顧客別営業戦略一覧（Sen-Ichi）」システムの詳細設計を記述したものです。  
開発者・保守担当者・新規参画エンジニアが本システムを正確に理解し、機能追加・障害対応・引き継ぎを円滑に行えることを目的とします。

## 1.2 対象読者

| 対象者 | 主な参照箇所 |
|---|---|
| 新規参画エンジニア | 全章 |
| フロントエンド開発者 | 3章（画面設計）・7章（共通仕様） |
| バックエンド開発者 | 5章（API設計）・4章（DB設計）・6章（認証） |
| インフラ・運用担当者 | 2章（システム概要）・8章（開発・運用） |
| プロジェクト管理者 | 1章・2章 |

## 1.3 適用範囲

本ドキュメントは以下のシステムに適用します。

- **システム名：** 顧客別営業戦略一覧（Sen-Ichi）
- **対象年度：** FY2026
- **リポジトリ：** `toshiki-ohuchi/Sen-Ichi-dev`
- **本番URL：** Vercel にデプロイされた本番環境

## 1.4 用語定義・略語集

| 用語・略語 | 説明 |
|---|---|
| SPA | Single Page Application。ページ遷移をブラウザ側で処理するWebアプリケーション形式 |
| SSR | Server Side Rendering。サーバー側でHTMLを生成する方式（本システムは非採用） |
| ORM | Object-Relational Mapper。SQLを直接記述せずにDBを操作できるライブラリ |
| Serverless Function | サーバーを常駐させず、リクエスト時のみ起動して処理するバックエンド実行形式 |
| 楽観的ロック | 同時編集の競合を更新時に検出する排他制御方式。本システムでは `version` カラムで管理 |
| session_id | ログイン後にブラウザのCookieに保存されるセッション識別子 |
| httpOnly | JavaScriptからアクセスできないCookie属性。XSS攻撃によるCookie窃取を防ぐ |
| ilike | PostgreSQLの大文字小文字を区別しない部分一致検索演算子 |
| Upsert | INSERT または UPDATE を条件に応じて自動で選択する操作 |
| SAS | エス・エー・エス（本システムを運用する組織名） |
| BP | ビジネスパートナー |
| FY | Fiscal Year（年度）。例：FY2026 = 2026年度 |
| admin | システム管理者ロール。ユーザー管理権限を持つ |
| user | 一般ユーザーロール。レコードの閲覧・編集のみ可能 |
