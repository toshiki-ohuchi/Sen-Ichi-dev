## 変更前セルフチェック（必須）

コードに変更を加える前に、**必ず以下のチェックを実施すること。**  
チェック完了後に変更に着手し、結果をステップ4の完了報告に含めること。

### チェック1：影響箇所の網羅的調査

変更対象のフィールド・関数・コンポーネント名を `grep` で全ファイル検索し、参照箇所をすべて列挙する。

確認対象：
- `src/` 内の Vue コンポーネント・ストア・型定義・API クライアント
- `api/` 内のルート・スキーマ・ミドルウェア
- `docs/詳細設計書/` 内の設計書（更新が必要な章の特定）

### チェック2：デグレリスク評価

各変更箇所について、以下の観点で「壊れる可能性があるもの」を列挙する。

| 観点 | 確認内容 |
|---|---|
| 画面表示 | 一覧・詳細・編集・閲覧モードそれぞれへの影響 |
| API | リクエスト・レスポンスの型変更による不整合 |
| DB | スキーマ変更による既存データへの影響 |
| Excel | エクスポート列順・インポート列インデックスへの影響 |
| 型定義 | `SalesRecord`・`SalesRecordInput`・初期値オブジェクトの整合性 |

### チェック3：実装方針の決定

変更に着手する前に、以下を自分で確認する。

- 変更するファイルと変更内容を具体的に列挙できているか
- 影響を受けるファイルをすべて変更リストに含めているか
- 判断に迷う箇所がある場合は、**実装前にユーザーへ確認する**

## 変更実施後の必須ルール

コードに変更を加えてコミット・プッシュした後は、**以下の手順を必ず一連で実行すること。**  
ユーザーから明示的に省略を指示された場合を除き、省略しない。

### ステップ1：コード変更をコミット・mainへマージ・プッシュ

作業ブランチに関わらず、**必ず最終的に `main` ブランチへマージしてプッシュすること。**

```bash
# 1. 変更をコミット（現在のブランチで）
git add <対象ファイル>
git commit -m "feat: ..."

# 2. mainへマージしてプッシュ
git checkout main
git pull origin main          # リモートの最新を取得（競合防止）
git merge <作業ブランチ名>
git push origin main
```

- mainブランチへのプッシュにより **Vercel が自動デプロイ** される
- featureブランチへのプッシュだけで終わらないこと（設計書もコードも必ずmainに届けること）

### ステップ2：DBスキーマ変更の案内（該当する場合のみ）
- `api/db/schema.ts` を変更した場合は、以下をユーザーに案内する：
  ```
  DBスキーマを変更しました。ローカル環境で以下を実行してください：
  npm run db:push
  ```
- `db:push` はローカルの `DATABASE_URL` が必要なため、自動実行はしない

### ステップ3：詳細設計書の更新
- 変更内容に関係する `docs/詳細設計書/` の章を特定する
- 対象章の判定基準：

| 変更箇所 | 更新する設計書 |
|---|---|
| `api/db/schema.ts` | `4_データベース設計.md` |
| `api/routes/*.ts` | `5_API設計.md` |
| `api/middleware/auth.ts` | `6_認証・セキュリティ設計.md` |
| `api/routes/auth.ts` | `6_認証・セキュリティ設計.md` |
| `src/views/*.vue` / `src/components/**` | `3_画面設計.md` |
| `src/types/index.ts` | `4_データベース設計.md` / `5_API設計.md` |
| `src/stores/*.ts` | `7_共通仕様.md` |
| `vercel.json` / `package.json` | `2_システム概要.md` |
| Excel入出力ロジック | `7_共通仕様.md` |
| 認証・セキュリティロジック | `6_認証・セキュリティ設計.md` |

- 該当章の内容をコードの変更内容に正確に合わせて修正する
- 修正した設計書を **コードとは別のコミット** でプッシュする（ステップ1と同じくmainへ）

### ステップ4：完了報告
- 実施した内容を以下の形式で報告する：
  ```
  【変更前セルフチェック結果】
  ■ 影響箇所：[調査で特定したファイル一覧]
  ■ デグレリスク：[リスクあり/なし、ありの場合は内容と対処]
  ■ 実装方針：[変更したファイルと変更内容の概要]

  ✅ コード変更：[変更の概要]
  ✅ Vercelデプロイ：mainプッシュにより自動実行
  ✅ 設計書更新：[更新したファイル名]
  （⚠️ DBスキーマ変更あり：npm run db:push を実行してください）

  【mainブランチ最新コミット確認】
  （git log origin/main --oneline -3 の出力を貼り付ける）
  ```
- 完了報告には必ず `git log origin/main --oneline -3` の実行結果を含めること
- コードと設計書の両方が `origin/main` に含まれていることをログで確認してから報告する

## 重要な実装仕様（変更時の注意点）

- **楽観的ロック**：`records.version` フィールドで管理。PUT時に `version` 不一致 → 409
- **セッション認証**：`sessions` テーブルで管理。Cookie名 `session_id`（httpOnly・7日間）
- **No自動採番**：`MAX(no) + 1` でサーバー側が採番。フロントから送らない
- **スケジュール保存**：Upsert（ON CONFLICT DO UPDATE）方式
- **TODO保存**：全DELETE → 全INSERT（replace方式）
- **契約形態**：複数選択を「・」区切りで1文字列に格納
- **セレクトボックス**：null/空文字の扱いは `:value="field ?? ''"` + `@change` パターン
- **modal-body**：`overflow-x: hidden` 必須（CSS sticky の横固定のため）
